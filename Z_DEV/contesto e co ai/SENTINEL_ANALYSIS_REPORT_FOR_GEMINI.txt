2. PROBLEMI CRITICI (da risolvere prima di continuare)
🔴 CRITICO-1 — naming_intelligence.py NON legge dalla Knowledge Base
File: core/naming_intelligence.py Problema: Tutti i pattern (suffissi, token, packed maps) sono hardcoded direttamente nel modulo in MAP_PATTERNS: dict = { ... }. La 06_KNOWLEDGE_BASE/mappings/naming_map.json esiste e contiene gli stessi dati, ma NamingIntelligence non la legge mai.


Violazione architetturale: REGOLE_GLOBALI dicono che la KB deve essere la single source of truth per naming. Attualmente ci sono 3 fonti concorrenti per gli stessi dati:


1. core/naming_intelligence.py — MAP_PATTERNS hardcoded
2. config/pipeline_config.yaml — sezione naming.vendor_patterns e naming.generic_patterns
3. 06_KNOWLEDGE_BASE/mappings/naming_map.json — generato da knowledge_processor.py


Fix richiesto: NamingIntelligence.__init__ deve accettare mappings_path e caricare i pattern da naming_map.json, mantenendo i valori hardcoded solo come fallback se il file non esiste.


# Come dovrebbe essere:


class NamingIntelligence:


    def __init__(self, mappings_path: str = "06_KNOWLEDGE_BASE/mappings/naming_map.json"):


        self.patterns = self._load_patterns(mappings_path) or MAP_PATTERNS

for f in files:


    parsed = parse_process_description(description, f.name)


    if parsed["process"] != "unknown":

Se importi una cartella con 10 texture e descrivi "normal da albedo con materialize", il process.json conterrà solo 1 entry per il primo file che matcha. Le altre 9 texture non avranno traccia del processo.

3. PROBLEMI STRUTTURALI (debt tecnico, non bloccanti ora)
🟡 STRUCT-1 — Compatibilità Python 3.8
Problema: Diversi moduli usano la sintassi list[str] e dict[str, ...] (lowercase built-in generics), disponibile solo da Python 3.9+.


File affetti:


* core/physics_oracle.py riga 114, 119
* core/asset_parser.py riga 77
* core/decision_layer.py righe 26, 48, 49
* core/variant_generator.py riga 60


Se il runtime di Gemini è Python 3.8, questi file crashano all'import.


Fix: Aggiungere from __future__ import annotations in cima a ogni file affetto, oppure usare from typing import List, Dict, Tuple e la sintassi List[str].

🟡 STRUCT-2 — 02_CUSTOM ha output senza sidecars

4. MIGLIORAMENTI RICHIESTI DAL NUOVO SPEC
📋 REQ-1 — Provider come dato esplicito, non cartella
Stato attuale: Il provider viene inferito dal nome della cartella genitore in 01_RAW_ARCHIVE/Provider/Material/.


Nuovo requisito: Il provider deve essere inserito esplicitamente durante l'import (tramite import_assistant) e salvato in material_info.json. Non più una convenzione di cartella.


Impatto: ingestor.py deve essere aggiornato per leggere provider da material_info.json in priorità rispetto all'inferenza da path. La logica _detect_provider() rimane come fallback.


________________


📋 REQ-2 — Anagrafica completa dalla fonte
Nuovo requisito: Quando si importa una texture da un sito fornitore (AmbientCG, Quixel, etc.), devono essere salvati nell'anagrafica:


* URL diretto alla risorsa
* Tag dal sito
* Tecnica di acquisizione (Photogrammetry, Procedural, Scanned)
* Categoria materiale dal sito
* Lista risoluzioni disponibili


Dove va: In material_info.json, scritto da import_assistant.


Dove lo usa: Il DatasetBuilder deve includerlo nel master.json output. Il NamingIntelligence può usare technique per selezionare regole di validazione appropriate (una texture Photogrammetry ha caratteristiche diverse da una Procedural).
